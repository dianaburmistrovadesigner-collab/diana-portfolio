## Набор функций MVP

Ниже — список фич/изменений, которые в заметках явно помечены как **“делать немедленно / Phase 1 (MVP) / Priority 1 (Critical)”**. Я включаю и юзабилити-правки.

### MVP / “делать немедленно” (1–2 месяца)

**1) Экстренное восстановление Dashboard (core usability + бизнес-контроль)**

- “Починить все графики: сломанные цвета, нерабочие фильтры, зависающие элементы”
- “Добавить названия API-ключей: заменить непонятные ID на человекочитаемые названия”
- “Показать баланс в деньгах: дублировать токены суммой в долларах”
- “Исправить сортировку: от большего к меньшему по затратам”
- “Внедрить кеширование: убрать 10-секундные загрузки”

**2) “Революционная аналитика по типам контента” (OUTPUT вместо моделей)**

- “Создать 4 основные категории: Текст, Изображения, Аудио, Видео”
- “Многоуровневая детализация: клик на категорию → топ моделей внутри”
- “Инсайты переливов: ‘текст падает, видео растет’ = смена трендов”

**3) Система критических алертов (финконтроль + безопасность)**

- “Алерты по балансу: 10%, 5%, 3%, 2%, 1% остатка”
- “Детекция аномалий: рост расходов >25% за день”
- “Push-уведомления: в браузере + email”
- “Настраиваемые пороги: под каждый бизнес”

### MVP / Phase 1 (MVP) — из заметки по ИА (веб-приложение)

(Там MVP описан как набор модулей/разделов, которые нужно реализовать первыми)

- “Templates — критично для всех сегментов”
- “Models — дублирование из документации”
- “Dashboard с калькулятором стоимости”
- “Analytics — расширение”

> Важно: в заметках **не зафиксировано**, что именно из “Templates/Models/Analytics” было уже реализовано как готовый MVP vs запланировано как MVP-этап разработки. Формулировка — “Phase 1 (MVP)”. Если тебе нужно в кейсе строго “что реально вошло в выпущенный MVP”, это **[нужно уточнить]**.

## Приоритизация

### Как приоритизировали (метод/критерии — как в заметках)

В явном виде в заметках встречаются такие критерии/основания:

- **По критичности для удержания (retention):**
    - “ROI: критично для retention — без работающего Dashboard менеджеры уходят”
- **По конкурентному преимуществу / уникальности:**
    - “ROI: конкурентное преимущество — такого нет ни у кого” (про категоризацию по типам)
- **По предотвращению прямых потерь пользователя:**
    - “ROI: предотвращение убытков пользователей” (про алерты и ботовые атаки)
- **По стадийности трансформации (“поэтапная трансформация”):**
    - “Сначала починить основу… Затем добавить уникальность… В конце строить экосистему…”

Отдельного названия метода (например, RICE/ICE/MoSCoW) в заметках нет — **[нужно уточнить]**.

## Обоснование решений

Почему эти изменения вошли в первую очередь — **по формулировкам из заметок**, в формате наблюдение → значение → дизайн-импликация.

### 1) Dashboard — “экстренно”

- **Наблюдение:** “Основной инструмент менеджеров полностью сломан”; “без работающего Dashboard менеджеры уходят”.
- **Значение:** это прямая причина потери аудитории, которая отвечает за контроль расходов/принятие решений.
- **Дизайн-импликация:** сначала восстановить базовую работоспособность и понятность (деньги вместо токенов, названия ключей, сортировка, скорость/кеширование), иначе остальные улучшения не удержат пользователей.

### 2) Категоризация по OUTPUT — “уникально и полезно”

- **Наблюдение:** “Инсайт от Юры (Project Manager): Категоризация по OUTPUT вместо моделей”; “такого нет ни у кого”.
- **Значение:** появляется конкурентное преимущество и более “управляемая” аналитика на уровне задач/типов генерации.
- **Дизайн-импликация:** строить аналитику и детализацию “сверху вниз”: 4 категории → проваливание в модели → отслеживание “переливов”.

### 3) Алерты и аномалии — “защита денег пользователя”

- **Наблюдение:** “Реальные кейсы: Сергей терял деньги из-за ботовых атак”.
- **Значение:** пользователи несут прямые финансовые потери без раннего предупреждения.
- **Дизайн-импликация:** встроить систему алертов (баланс/аномалии) + push/email + настраиваемые пороги.

### 4) Почему “Playground v2.0” и “Community” — не приоритет

- **Наблюдение:** “Разработчики не используют, менеджеры редко” (про Playground v2.0); “Discord практически не используется” (про community).
- **Значение:** эти направления не закрывают самые критичные боли сейчас (dashboard/контроль/выбор) и не дают сравнимого ROI.
- **Дизайн-импликация:** сфокусироваться на фундаменте и бизнес-контроле; улучшения Playground — позже и целевым образом (скорее для неразработчиков).
