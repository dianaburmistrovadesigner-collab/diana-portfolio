## Шаги каждого flow

### Flow 1 — Startup создает AI‑сервис CourseGenie (James + Tom)

Источник: таблица “User Flow 1 - Startup creates AI service CourseGenie using AIMLAPI”.

**1. Экран:** AIMLAPI homepage  

- **Действие:** James studies pricing information and capabilities
- **Ветвление:** Moving to competitors, closing the site
- **Ключевая функция/интерфейс:** Cost calculator for startups with load distribution settings
- **Комментарий/барьер:** Startups are very price-sensitive, need to show savings immediately

**2. Экран:** Technical documentation  

- **Действие:** Tom studies API capabilities and integration examples
- **Ветвление:** Moving to other providers, searching for alternatives
- **Ключевая функция/интерфейс:** Comparative table of AI providers with demo sandbox
- **Комментарий/барьер:** Tech leads make the final decision, need concrete examples

**3. Экран:** Comparison page  

- **Действие:** Team analyzes advantages vs. competitors
- **Ветвление:** Choosing another provider, postponing decision
- **Ключевая функция/интерфейс:** Detailed feature comparison with savings calculator
- **Комментарий/барьер:** Need to clearly show advantages, avoid feature parity

**4. Экран:** ROI calculator  

- **Действие:** James calculates economics: $300 vs $800
- **Ветвление:** Budget revision, searching for cheaper solutions
- **Ключевая функция/интерфейс:** ROI calculator with cost growth forecasting
- **Комментарий/барьер:** Economic justification is critical for decision-making

**5. Экран:** Template library  

- **Действие:** Tom searches for ready-made solutions for EdTech
- **Ветвление:** Creating from scratch, searching for other templates
- **Ключевая функция/интерфейс:** Template catalog with filtering by industries and preview
- **Комментарий/барьер:** Ready-made templates are a key advantage, need quick search

**6. Экран:** EdTech template page  

- **Действие:** Team chooses Educational Content Generator
- **Ветвление:** Choosing another template, custom development
- **Ключевая функция/интерфейс:** One-click deployment with capability description
- **Комментарий/барьер:** First impression of the product, should be impressive

**7. Экран:** Test environment  

- **Действие:** Tom tests functionality in 30 minutes
- **Ветвление:** In-depth testing, rejecting the solution
- **Ключевая функция/интерфейс:** Stable operation with fast API response (<2 sec)
- **Комментарий/барьер:** Setup speed is a competitive advantage

**8. Экран:** Coverage dashboard  

- **Действие:** James evaluates 60% MVP coverage
- **Ветвление:** Searching for additions, changing solution
- **Ключевая функция/интерфейс:** Coverage metrics with visualization and recommendations
- **Комментарий/барьер:** Helps decide whether to continue work

**9. Экран:** Code Editor  

- **Действие:** Tom programs unique functions
- **Ветвление:** Using only ready-made, hiring a developer
- **Ключевая функция/интерфейс:** Code editor with prompt library and A/B testing
- **Комментарий/барьер:** Customization is key to product uniqueness

**10. Экран:** Billing settings  

- **Действие:** James configures limits and rates
- **Ветвление:** Postponing monetization, using free version
- **Ключевая функция/интерфейс:** Limit management system with payment integration
- **Комментарий/барьер:** Monetization should be simple for non-technical users

**11. Экран:** Integration  

- **Действие:** Tom integrates AI into EduFlow platform
- **Ветвление:** Using via API, creating separate service
- **Ключевая функция/интерфейс:** SDK for React/Vue with detailed documentation
- **Комментарий/барьер:** Integration simplicity is critical for developers

**12. Экран:** Beta testing  

- **Действие:** Team launches test with 50 users
- **Ветвление:** Public launch, expanding beta
- **Ключевая функция/интерфейс:** Staging environment with beta testing management tools
- **Комментарий/барьер:** Critical stage of product validation

**13. Экран:** Analytics dashboard  

- **Действие:** Tom optimizes AI usage
- **Ветвление:** Ignoring optimization, manual configuration
- **Ключевая функция/интерфейс:** Dashboard with cost optimization recommendations
- **Комментарий/барьер:** Cost optimization is critical for startups

**14. Экран:** Business analytics  

- **Действие:** James tracks unit economics
- **Ветвление:** Intuitive management, external tools
- **Ключевая функция/интерфейс:** Integration with Google Analytics, ready-made reports for investors
- **Комментарий/барьер:** Unit economics must be transparent

**15. Экран:** Scaling  

- **Действие:** Team grows to 500 users
- **Ветвление:** Growth limitation, migration to another solution
- **Ключевая функция/интерфейс:** Automatic scaling with enterprise features
- **Комментарий/барьер:** Growth should be painless

**16. Экран:** Investor presentation  

- **Действие:** James demonstrates AI economics
- **Ветвление:** Self-financing, searching for other investors
- **Ключевая функция/интерфейс:** Ready-made report templates with ROI calculator
- **Комментарий/барьер:** Will help attract investments

**Ключевые точки решений (из “Key Decision Points”):**

- Critical forks:
    
    1) Steps 1-4: Economic justification  
    
    2) Steps 5-6: Availability of suitable templates  
    
    3) Steps 7-8: Setup speed and requirements coverage  
    
    4) Step 12: Beta testing results  
    
- Main barriers: Price sensitivity; Technical complexity of integration; Setup time; Documentation quality for developers
- Success factors: Economic advantages ($300 vs $800); Ready-made templates; Integration simplicity via SDK; Scaling support

### Flow 3 — Текущий flow выбора и внедрения API‑сервиса (current user flow)

Источник: блок “🚀 Полный процесс выбора и внедрения API сервиса (current user flow)”.

**1. Инициация потребности**  

- Кто: Product Manager/CTO
- Что: появляется задача добавить AI‑функционал в продукт

**2. Первичный исследовательский этап**  

- Кто: менеджер (не разработчик)
- Что делает: гуглит, изучает топ‑решения, сравнивает общие возможности; “ищет именно модели под свое решение, но берет с запасом”
- Ключевая проблема: “Часто не знает, что может потребоваться завтра”

**3. Тестирование в Playground**  

- Кто: менеджер
- Что происходит: создает аккаунт, примерно выбирает модель, делает тестовые запросы
- Цель: понять “работает ли вообще для наших задач, как работает API в целом”

**4. Техническая растерянность**  

- Кто: менеджер пытается оценить техническую состоятельность
- Проблема: “Ничего не понимает в технических деталях”
- Решение: зовет разработчика

**5. Разработчик входит в игру**  

- Кто: разработчик
- Что делает: пробегается по маркетинговому сайту, “сразу идет в документацию”

**6. Критический момент — качество документации**  

- Ключевой инсайт: “Качество документации = первый и самый критичный критерий отбора”
- Правило: “Если документация плохая → сразу закрывают и уходят навсегда”

**7. Генерация тестового API‑ключа**  

- Кто: разработчик
- Что: идет в личный кабинет, создает ключ для первичной оценки API

**8. Техническое тестирование**  

- Кто: разработчик
- Как: тестирует через прямые запросы (curl, Postman), “НЕ через Playground”

**9. Консолидация и подготовка к согласованию**  

- Кто: менеджер + разработчик
- Что делают: собирают плюсы/минусы, готовят презентацию
- Проблема: “Цены непонятны → берут самый дешевый тариф на месяц”

**10. Согласование с руководством**  

- Кто: руководитель
- Решение: дает добро на тестовое внедрение

**11. Создание API‑ключа (боевой/именованный/с ограничениями)**  

- Кто: разработчик
- Критично: “Безопасность — менеджер контролирует доступ и бюджет”
- Что создает: “именованный ключ с ограничениями или отдельный ключ для разработчиков”

**12. Техническая интеграция и продуктовый релиз**  

- Кто: разработчик (интеграция)
- Важная связка: “API‑ключ контролирует менеджер (чаще всего НЕ разработчик!) (безопасность, бюджет)”

**13. Попытки контроля бюджета**  

- Кто: менеджер
- Реальность: “Dashboard сломан, данные непонятны”
- Проблема: “Нет связи между ключами и реальными тратами”

**14. Мониторинг качества и стоимости**  

- Кто: менеджер + разработчик
- Что отслеживают: реальные расходы vs ожидания, качество результатов

**15. “Скорбная” оптимизация**  

- Кто: менеджер
- Что: пытается оптимизировать расходы постфактум по статистике трат

**Ключевые болевые точки (как в заметках):**

- “Критическая точка отсева”: шаг #6 — качество документации
- “Финансовая слепота”: шаги #9, #12 — “никто не понимает реальную стоимость”
- “Новая критическая точка”: шаг #11 — управление ключами, командный доступ
- “Синдром неопределенности будущего”: шаг #2

## Что изменилось vs текущий flow

В заметках есть “сравнение” в виде явного описания **текущего процесса (current user flow)** и предполагаемого нового подхода (в виде дизайна flow/концепта и “рекомендаций/решений”). Собираю только то, что прямо описано.

### 1) Playground и тестирование: “как было”

- **Было (current user flow):**
    - Менеджер использует Playground на раннем этапе (Step 3), чтобы понять “работает ли вообще”.
    - Разработчики “НЕ через Playground” — тестируют через curl/Postman (Step 8), потому что часто игнорируют playground.
- **Стало (в flow-сценариях и целевых шагах):**
    - В Flow 2 (Sofia): Playground — “first product usage experience” и точка решения (Step 7-8), причём предполагается “Playground with ready templates for Amazon”.
    - В Flow 1 (Startup): тестирование вынесено в “Test environment” (Step 7) и связано со скоростью (<2 sec) и валидацией покрытия (Coverage dashboard Step 8).
    - Явного текста “Playground было таким → стало таким” нет; но есть смена роли: от “необязательного для dev” к “шагу первичной ценности для новичка” в сценарии Sofia. **[нужно уточнить]** — была ли это часть реально сделанного редизайна или концепта.

### 2) Выбор модели и “готовый старт”: “как было”

- **Было (current user flow):**
    - Менеджер на раннем этапе “примерно выбирает модель” (Step 3), при этом есть “синдром неопределенности будущего” (Step 2).
    - Разработчик оценивает по документации; если документация плохая — уход навсегда.
- **Стало (в flow-сценариях):**
    - Flow 1: старт через “Template library” → “One-click deployment” + “requirements coverage (60% MVP)” (Steps 5-8).
    - Flow 2: старт через калькулятор → быстрый тест → “No-code builder” + готовые Amazon‑шаблоны с валидацией (Steps 5-10).

### 3) Экономика и ROI: “как было”

- **Было (current user flow):**
    - На шаге 9 “Цены непонятны → берут самый дешевый тариф на месяц”.
    - Далее “финансовая слепота” и “dashboard сломан”.
- **Стало (в flow-сценариях):**
    - Flow 1: Cost calculator + ROI calculator + forecasting (Steps 1,4).
    - Flow 2: calculator (1000 descriptions + 500 images = $80), сравнение vs конкуренты и “money-back guarantee” (Step 5).

### 4) Роли и ответственность за ключи: “как было”

- **Было (current user flow):**
    - Менеджер контролирует доступ и бюджет, разработчик интегрирует (Step 11-12).
    - Нужен workflow передачи доступов и командный доступ (отмечено как критическая точка).
- **Стало:**
    - В самих User Flow 1/2 это не развернуто как “было → стало”, но в текущем запросе “User Flows” это в виде шагов не описано. **[нужно уточнить]**, где именно в flow это отразилось (в других заметках это вынесено в стратегию/ИА/ролевую модель, но ты просила именно flows).
